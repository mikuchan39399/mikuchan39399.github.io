Linux信号与中断

Ctrl + C 干了什么

平时我们运行一些死循环程序时，可以在命令行终端上用Ctrl + C 快捷键结束进程。
实际上，这个快捷键让键盘外设控制 CPU 让操作系统往目标进程发送SIGINT信号以杀死进程，这里有一个 API 能让我们验证一下...

#include <signal.h>
typedef void (*sighandler_t)(int);
sighandler_t signal(int signum, sighandler_t handler);
  • 作用：修改目标信号的处理回调函数
  • 参数
    • singnum 信号编号
    • handler 信号处理函数
  • 返回值 修改之前的旧信号处理函数的函数指针
#include <iostream>
#include <csignal>
void handler(int signo)
{
    std::cout << "捕捉到" << signo << "号信号\n"; 
}
int main()
{
    signal(SIGINT, handler);
    while(1);    
    return 0;
}

我用这个接口捕捉了SIGINT信号，让我们看看现在用Ctrl + C还能不能杀掉这个进程

当使用Ctrl + C杀不掉某个进程的时候，一种可能是程序捕捉了SIGINT，还有可能是这个进程不是前台进程，而前台进程只有一个，键盘只能给前台进程发信号，⼀个命令后⾯加个&可以放到后台运⾏,这样 Shell 不必等待进程结束就可以接受新的命令启动新的进程。

信号机制

可以用kill -l 命令查看所有信号

为什么没有32, 33号信号？貌似是因为某些历史遗留原因

每个信号名都有一个宏，表示信号名对应的信号编号
编号34以上是实时信号，咱们在这儿不讨论实时信号，可以用man 7 signal查询各个信号的产生条件和默认处理动作
信号的默认处理动作有三种：

  • 忽略 SIG_IGN
  • 默认 SIG_DFL
  • 捕捉，提供相应自定义信号处理函数

其实SIG_DFL和SIG_IGN就是把0,1强转为函数指针类型

信号产生

信号产生有以下几种方式

  1. 外设，例如刚才键盘上的 Ctrl + C
  2. 命令行工具：kill -信号名 pid 可以往指定进程发送对应信号
  3. 程序内函数：
#include <sys/types.h>
#include <signal.h>
int kill(pid_t pid, int sig); // 往指定进程发送信号

#include <signal.h>
int raise(int sig); // 往当前进程发送信号

#include <stdlib.h>
void abort(void); // 让当前进程接受信号而 <异常> 终止
// 就像exit函数⼀样,abort函数总是会成功的,所以没有返回值。
  1. 软件产生信号
#include <unistd.h>
unsigned int alarm(unsigned int seconds);
  • 作用：设定一个闹钟，让内核在 seconds 秒后给当前进程发送 SIGALRM
  • 返回值：
    • 此闹钟之前没有运行时闹钟，返回 0
    • 否则返回运行时闹钟剩余触发秒数并重置闹钟状态为当前调用
  1. 硬件异常产生信号
    硬件因为某些原因产生异常后，被硬件以某种方式检测到并通知操作系统内核，让内核向当前进程发送适当信号，例如：
  • 除以0 - CPU 上的 ALU 运算单元产生异常，操作系统发送SIGFPE信号
  • 野指针/空指针/非法内存地址 - MMU内存管理单元产生异常，操作系统发送SIGSEGV信号

信号传递

实际执行信号处理动作被称为信号递达Delivery
信号从产生到递达之间的状态，被称为信号未决Pending
信号可以被进程选择性阻塞
被阻塞的信号会一直处于未决状态，直到进程解除对此信号的阻塞，才能信号递达
阻塞不同于忽略，忽略本身就是一种信号递达
每个进程当中都有两个位图和一个函数指针数组，分别为：

  • block 位图
  • pending 位图
  • handler sighandler_t类型函数指针数组
    这些位图的数据类型是sigset_t，被称作信号集，block信号集也被称为信号屏蔽字
    信号编号就是这两个信号集和函数指针数组的下标，SIG_IGN和SIG_DFL要被强转成sighandler_t类型放进这个数组中
    这两个信号集用于标识进程中每个信号的状态
    有一些可以操纵信号集的函数：
#include <signal.h>
int sigemptyset(sigset_t *set); //清空信号集为全0
int sigfillset(sigset_t *set);  // 填满信号集为全1
int sigaddset(sigset_t *set, int signo); //将信号集的目标信号编号标志位置为1
int sigdelset(sigset_t *set, int signo); // 将信号集的目标信号编号标志位置为0
int sigismember(const sigset_t *set, int signo); // 判断目标信号标志位在信号集是否为1
int sigpending(sigset_t *set); // 读取当前未决信号集，通过set输出型参数传出

比较重要的一个：

#include <signal.h>
int sigprocmask(int how, const sigset_t *set, sigset_t *oset);

作用：读取或更改进程的信号屏蔽字/阻塞信号集
参数：

  • set 作用于进程的信号屏蔽字
  • oset 之前的信号屏蔽字
  • how
    • SIG_BLOCK 按位或
    • SIGUNBLOCK 按位与 (~set)，解除阻塞的信号
    • SIG_SETMASK 覆盖

以下代码演示了阻塞的信号只能待在未决信号集里吃灰

#include <iostream>
#include <csignal>
#include <unistd.h>
#include <functional>
#include <sys/wait.h>
#include <vector>
#include <sys/types.h>

void PrintPending(sigset_t& pending)
{
    std::cout << "\n[ sigpending list: ";
    for(int i = 31; i; i--)
    {
        if(sigismember(&pending, i))
        {
            std::cout << "1";
        }
        else
        {
            std::cout << "0";
        }
    }
    std::cout << " ]" << "\r";
}

int main()
{
    sigset_t block, old_block;
    sigemptyset(&block); sigemptyset(&old_block);
    sigaddset(&block, SIGINT);
    sigprocmask(SIG_SETMASK, &block, &old_block);
    while(true)
    {
        sigset_t pending;
        sigemptyset(&pending);
        sigpending(&pending);
        PrintPending(pending);
        sleep(1);
    }
    return 0;
}

SIGINT信号被阻塞之后就算不捕捉它，也无法到达信号递达的状态，也就退出不了进程了

捕捉信号也可以用一组oop一些的接口：

#include <iostream>
#include <unistd.h>
#include <signal.h>

void handler(int signo)
{
    std::cout << "获取到一个信号: " << signo << std::endl;
}

int main()
{
    struct sigaction act, old_act;
    act.sa_handler = handler;
    act.sa_flags = 0;
    sigemptyset(&(act.sa_mask));
    sigaction(SIGTERM, &act, &old_act);
    while(1)
    {
        std::cout << "我是一个进程: " << getpid() << std::endl;
        sleep(1);
    }
}

这个sa_mask成员比较有意思
这个信号集让对应置1信号被处理时自动变为阻塞状态，防止信号被递归处理

信号处理

信号处理是异步的！
也就是说，一个进程收到信号的时候，他不一定会马上去执行对应的handler函数
那它什么时候才去执行？

  1. 此时程序处于用户态愉快地执行着，此时因为神秘的原因程序进入内核态
  2. 程序现在处于内核态，处理完对应任务之后，顺手检查一下看看有没有信号需要处理的，如果没有捕捉信号采用自定义处理办法，忽略和默认行为操作系统知根知底，顺手执行完后返回用户态原先的位置继续执行下去
  3. 如果用户自定义了信号处理函数，万一这个函数有问题怎么办，可不敢让它在内核态执行，这里什么权限都有，操作系统要回到用户态执行这个函数
  4. 回到用户态，此时程序去执行handler自定义函数，然后返回内核态，因为在用户态找不到之前程序执行的位置
  5. 内核态中有一个sys_sigretun()函数，拥有足够的权限让程序返回用户态后从主控制流程中上次被中断的地方继续往下执行

这就是一个无限形的流程，节点在内核态

中断---操作系统到底是怎么运行的

在 CPU 日常运行各种程序的时候，鼠标点了一下，键盘按了一下，网卡收到了数据，等等不受预期的行为，都会打断 CPU 的动作，迫使 CPU 放下手中的事情进入内核态去执行其他任务，这种行为被称作中断，也是操作系统本身能够运作的重要基础

硬件中断/外中断

在键盘上按个快捷键是怎么让操作系统给进程发信号的？
这就是一种硬件中断！
我们电脑上的各种外设设备，都连接着一个名叫中断控制器的硬件结构，用来管理杂乱的外设请求。操作系统中存在着一个名叫中断向量表IDT的指针数组，在操作系统开机时就存在于内存中
一次完整的硬件中断的流程是这样的：

  1. 外设就绪：键盘被按下，鼠标被点击
  2. 发起中断：外设向中断控制器发起中断，中断控制器管理好外设的中断请求，决定优先级
  3. 通知 CPU：CPU 上的 INTR引脚接收中断控制器发来的高电平信号
  4. CPU 回应：在 CPU 回应中断控制器时，中断控制器把这个中断对应的中断向量号（一个 8 位的整数编号）通过数据总线扔给 CPU
  5. CPU 保护现场：保存中断前的上下文，以便中断结束后继续执行任务
  6. CPU 陷入内核态，根据中断号，去中断向量表中查表，执行对应的中断处理程序ISR，以回应不同的中断需求
  7. CPU 执行完毕，恢复现场中断结束，继续执行之前的工作

时钟中断

之前提到过，进程检查信号需要 CPU 先陷入内核态，那不能没有外设就绪就不检查信号了吧？
操作系统来调度进程，可是谁来调度操作系统自己？
CPU 中内置了一块称作时钟源的硬件，它可以周期性地以固定时间间隔不断通过中断向量表或可编程间隔定时器，向 CPU 发起中断，这被称作时钟中断

Note

CPU 主频 和 时钟中断频率并不等价
CPU 主频 = 外频（时钟源频率 / BCLK） × 倍频（Multiplier）
时钟源又叫晶振，只要给它通电，它就会产生极其稳定、规律的电脉冲信号，通常不高，比如 100MHz
时钟源的作用就是给整个主板上的各个组件提供一个统一的“基准时间尺度”，确保大家在一个节拍下协同工作，有点像乐队中的鼓手
CPU 内部集成了一个名叫 锁相环PLL的电路，他能放大n倍时钟源频率，这之后才是 CPU 的主频
时钟中断并不是 CPU 自己凭空产生的。主板上还有一个专门的可编程硬件定时器PIT 或 APIC Timer，它同样接收晶振脉冲信号。
定时器会根据设定好的计数值不断倒数，每倒数到 0，就会向 CPU 发送一次硬件中断信号IRQ 0。
操作系统在启动时，会配置这个定时器的倒数周期，从而决定了时钟中断频率（例如 Linux 内核中的 HZ 宏，通常配置为 250Hz 或 1000Hz）。

Important

CPU 主频决定了指令执行有多快，而时钟中断频率由操作系统规定

有了周期性的时钟中断，CPU 就能每隔一段时间自动陷入内核态，完成基本的信号检测
而时钟中断对应的中断向量，也就是对应时钟中断的中断处理程序，就是完成操作系统基本生命活动的代码集合

Important

操作系统的本质：一个由中断驱动的“死循环”
操作系统在开机初始化完毕后，其实就进入了一个静静等待事件发生的无限循环。它能够持续掌控全局，不被任何跑飞的用户程序独占 CPU，最核心的底气就来源于周期性的时钟中断。
硬件定时器就像维持系统生命特征的“心跳”，每隔一段固定时间就强制触发一次中断。这迫使 CPU 暂停当前正在执行的用户程序，自动陷入内核态。
随后，CPU 去中断向量表中寻找时钟中断对应的ISR并执行。这段内核代码承载了操作系统最基础的生命活动：它不仅会顺手检查并处理当前进程的未决信号，还会更新系统时间、扣减当前进程的时间片，并在时间片耗尽时触发进程调度。没有时钟中断，整个操作系统的运转就会陷入停滞。

void main(void) /* 这⾥确实是void，并没错。 */
{ /* 在startup 程序(head.s)中就是这样假设的。 */
...
/*
* 注意!! 对于任何其它的任务，'pause()'将意味着我们必须等待收到⼀个信号才会返

    for (;;)

CS寄存器

CPU 通过 CS 寄存器（代码段寄存器），来得知自己目前处于内核态还是用户态
CS 寄存器是一个 16bit 寄存器，最低的两位被称为CPL，当前特权级

  • CPL = 00 内核态
  • CPL = 11 用户态
    用户态程序妄想修改CPL寄存器，会被操作系统发送SIGSEGV段错误信号杀掉
    但是！我程序还在用户态的话，无论如何都不可能通过程序来修改CPL寄存器了呀？
    所以，修改CPL从3到0的操作绝对是由 CPU 的硬件电路全自动完成的！
  1. 中断向量表中除了存着内核代码的地址，还有一个预设好的内核代码段选择子，包含目标特权级 DPL = 0
  2. 在执行中断处理程序前，CPU 会用安保硬件电路核对权限之后，直接用物理电路把这个 特权级0的段选择子拍进 CS寄存器中
  3. 此时 CPU 已经陷入内核态，自然就能执行内核的中断处理程序了
    CPL 从0到3，回到用户态的时候倒是靠软件指令，iret，它可以把当初压在内核栈中的 CPL = 3旧 CS寄存器值重新写回 CS寄存器

内中断

内中断，又叫做异常，相比于异步的外中断，是同步触发的，分为这三类：

  • Trap 陷阱/软中断
  • Fault 故障
  • Abort 终止

Trap

在代码中主动执行的，能让 CPU 陷入内核的指令，被称作 Trap
一般有 syscall int 0x80 int 3 这三种
在32位机器上，一般使用 int 0x80，意思就是，去执行中断向量表中第 128号 中断向量对应的代码
在64位机器上，一般使用 syscall，它不去查中断向量表，而是直接去模型特定寄存器MSR中读出代码地址，更快

揭秘系统调用的幕后
我们平时用的系统调用，实际上并不是真正的系统调用接口，而是C语言封装好的系统调用库
内核中存在一个系统调用表，也就是函数指针数组，它们指向真正的系统调用
在我们调用系统库函数时：

  1. C库会帮我们找到系统调用名对应的系统调用号，也就是那个指针数组下标
  2. 把这个系统调用号写入通用寄存器中，mov eax, n
  3. 然后调用 syscall 或 int 0x80 陷入内核态
  4. 对应的中断执行代码则会拿到寄存器中的系统调用号，去查系统调用表并执行

所以说系统调用的实现借助了 Trap 软中断

调试器也用到了软中断
int 3 是一条用于代码调试的特殊汇编指令，用于执行IDT中的3号中断处理程序

  1. 打断点
    在代码的某一行打上一个断点时，调试器，比如GDB，会记录那行代码原来的第一字节机器码，然后用一个特殊的单字节指令0xCC把原来那个字节覆盖掉，这个特殊的指令在x86架构的汇编助记符中，就是int 3
  2. 程序在用户态欢快运行时，只要程序计数器碰到了这个单字节指令，CPU 就会陷入内核态，去执行3号中断处理程序
  3. 此时 CPU 执行3号中断处理程序，操作系统会检查到，这个是由用户态主动触发的断点异常，然后会往当前进程发送一个SIGTRAP信号，然后这个进程就被挂起了，不再参与CPU竞争。
    然后操作系统会通知早有预谋的调试器（它通常调用了一种ptrace系统调用），调试器就可以看到程序目前的状态机，并展示在IDE上，咱们接下来就能按F10/F11进行单步调试
  4. 继续执行跳过断点时，调试器就会把记录的真正的一字节机器码重新写回内存覆盖掉那个0xCC，程序计数器回退一个字节，重新执行断点所在行的指令，就好像什么都没发生一样

Fault

如果说 Trap 是程序主动，CPU 以比较体面的形式陷入内核
那么 Fault 就不是很体面了，这通常是由于 CPU 硬件遇到了突发问题，无法正常执行，被动强行中断。

Note

关于 CPU 保存现场，硬件中断/陷阱 与 故障和终止有一些区别

  • 特权级是否切换

    • 用户态陷入内核态
      用户栈的栈顶指针应该压入内核栈保存起来，也就是 SS 和 ESP 寄存器，然后再压入 EFLAGS、CS 和 EIP。
      此时保存的 CPU 上下文是用户态上下文，被打包成 pt_regs 压在当前进程内核栈空间的最高地址处。能通过 task_struct 中的 stack 指针找到它：stack 指针保存的是当前进程专属内核栈空间的最低地址，通过 stack + 当前进程内核栈总大小 - sizeof(pt_regs) 这种偏移量计算就能精准定位到 pt_regs。

    • 内核态继续留在内核态
      比如内核正在处理硬盘中断，突然网卡发来数据，发生中断嵌套。内核本来就在内核栈了，自然不用压入用户态栈顶指针，直接往内核栈压入 EFLAGS、CS 和 EIP 就行了。
      如果此时发生了进程调度切换，操作系统会将 ESP 和 EIP 存入 task_struct 内部的 thread_struct 里，以便下次被调度时能从原位置复活。此时 CPU 保存的是内核态上下文，它保存在内核栈深处。

  • 是否携带错误码 (Error Code)

    • 外部硬件中断和陷阱：没有错误码。
    • 某些严重的内部故障或终止：携带错误码。压完 EFLAGS、CS 和 EIP 之后，还会额外压入一个 Error Code 进内核栈，这是给操作系统法官判案看的。

  • 你要回哪去 (EIP 的指向)
    CPU 压入内核栈的 EIP 程序计数器地址并不相同：

    • 硬件外部中断和陷阱中压入的 EIP 是下一条即将执行的指令。
    • 故障压入的 EIP 必须死死指着发生错误的当前这条指令本身，因为有些故障是能抢救回来的（比如缺页中断）。

  • 中断之前的通用寄存器呢？（比如 EAX、EBX、ECX）
    通用寄存器的保存不归 CPU 硬件管，中断处理程序的开头会调用 pusha 或 pushad 等指令，把这些寄存器全压入内核栈。

除0故障

  • 硬件电路的崩溃
    1. CPU 在 程序计数器中拿到了这条除法指令
    2. 算术逻辑单元ALU拿到操作数，准备做除法，可是除0根本算不出来
    3. 硬件流水线会自己停下，这是纯硬件行为，不需要操作系统介入
  • 硬件自救
    1. CPU 会保护现场，把当前的各种状态寄存器EFLAGS，CS寄存器，程序计数器压入内核栈
    2. CPU 强行修改 CPL 为0陷入内核态
    3. 硬件设计规定死了，除0故障就要去执行中断向量表中的0号中断处理程序，CPU 直奔0号门
  • 操作系统判决
    1. 操作系统知道除0故障无法补救，直接找到当前进程PCB，给当前进程发送SIGFPE信号

各种算数异常通通叫做浮点异常，这是从古老的Unix时代一直沿用至今的

  • 进程死亡过程
    1. 检查未决信号集时发现表里挂着一个SIGFPE信号...
    2. 这个信号的默认动作就是终止进程

如果程序不要脸捕获了这个信号的话，因为 CPU 的状态机没变，操作系统就会一直往这个进程发送SIGFPE信号

  1. 进程：那还说啥了我死了得了呗。操作系统回收资源，终端上打印出一行字Floating point exception (core dumped)

野指针/空指针故障
假设你的 C++ 代码写了这么一句作死的操作：int* p = nullptr; *p = 39;

  • 查页表
    1. CPU 在 EIP 寄存器中拿到了这条赋值指令，准备把42写进虚拟地址0x0
    2. CPU 内部的 MMU 拿到虚拟地址0x0，去CR3寄存器找到页表的物理地址然后去查页表
  • 硬件碰壁
    1. MMU 发现虚拟地址0x0对应的页表项，其有效位 Present Bit 是0
    2. CPU 会把这个虚拟地址0x0 保存在CR2寄存器/页故障线性地址寄存器中
    3. CPU 硬件流水线立刻刹车，CPU 保护现场的同时额外压入一个Error Code，然后硬件自己强制跳入IDT的第14号门，也就是缺页异常对应的中断处理程序，CPU 要靠操作系统法官来判别此次故障的性质
  • 操作系统判断故障
    1. 操作系统去读CR2寄存器，获得异常虚拟地址0x0，并且结合内核栈中压入的Error Code，操作系统得知这是一次用户态下的写操作引发的缺页故障
    2. 操作系统去当前进程的task_struct中找到了一棵红黑树，这颗红黑树管理了此进程中所有合法的vm_area_struct
    3. 操作系统拿着0x0地址在这颗红黑树中比对，原来这个虚拟地址是个非法的！现在连缺页故障都算不上了，此进程应当被判处死刑
  • 无情击毙
    1. 操作系统给当前进程发送一个人民喜闻乐见的SIGSEGV段错误信号
    2. CPU 准备返回用户态时，发现未决信号集中的SIGSEGV，执行默认操作强制终止进程，在终端上打印Segmentation fault (core dumped)

Abort

一般触发Abort异常的话，大概率是因为硬件和核心数据已经没救了，操作系统调用panic()函数，触发内核恐慌Kernel Panic，全系统死机或者重启，在Windows中，这就是蓝屏死机了

Note

信号处理：从内核态执行用户态 Handler 的复杂内幕

为了让用户态的普通 handler 函数可以正常执行，并在结束后把控制权平滑交还给被中断的主程序，操作系统会在切回用户态之前，对当前进程的用户栈做一套极其精妙的“小手脚”：

  1. 保存现场 (内核栈)：程序最初被打断并陷入内核态时，CPU 硬件和操作系统的中断处理程序，已经将主程序的全部硬件上下文（原程序计数器 EIP、ESP、EFLAGS、各种通用寄存器等）安全地保存在了内核栈中。
  2. 构建信号栈帧 (备份至用户栈)：为了在 handler 执行完后能找回回家的路，内核会提取内核栈中保存的上下文快照 ucontext，并将其压入用户栈。同时，内核会在用户栈上布置好 handler 需要的参数（如信号编号 signo）。
  3. 压入跳转代码：内核继续在用户栈的栈顶压入一个伪造的返回地址。这个地址并不指向主程序的原位置，而是指向一段用于触发 sigreturn 系统调用的特殊跳转代码（也就是常说的蹦床）。
  4. 篡改返回路线 (修改内核栈快照)：内核直接去修改此刻仍保存在内核栈上的寄存器快照，把即将被恢复的 EIP 强行覆写为 handler 函数的入口地址，并把 ESP 覆写为刚刚布置好的用户栈新栈顶。
  5. 降级跃迁 (重返用户态)：内核执行 iret 指令，CPU 从内核栈中弹出被“篡改”过的状态，特权级降回 CPL=3。此时程序完美降落回用户态，毫不违和地开始执行 handler，而它操作的正是刚刚被内核精心伪造过的用户栈。
  6. sigreturn：当 handler 遇到 return 时，它会从用户栈弹出那个伪造的蹦床地址并跳过去。蹦床代码顺势执行 syscall 或 int 0x80 再次陷入内核。此时内核执行 sys_sigreturn，去用户栈里把步骤 2 中备份的 ucontext 挖出来，直接覆盖到硬件寄存器上。最后再执行一次 iret，主程序就在原地址满血复活了
