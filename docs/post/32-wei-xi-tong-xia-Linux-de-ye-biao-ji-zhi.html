32位系统下Linux的页表机制

在32位系统下，每个进程拥有 4GB 的虚拟地址空间
在Linux中，与内存进行IO的最小单位是4KB，在文件系统中是一个块的大小，在物理内存中是一个页框的大小，虚拟文件系统把磁盘空间划分为一个一个4KB的块，分页式管理把物理内存划分为一个一个4KB的页框，页表映射的，也不是字节号，而是页框编号

页目录

在32位系统中，内存中存在一个名叫页目录的页框，它的物理地址存在于CR3寄存器中，页目录4KB的大小中有着1024个页目录项PDE，其中存放着1024张页表的物理页框号PFN + 标志位，因为一张张页表本身也是一个个页框，所以这样找到对应的页表的物理内存后，也就能去通过页表中的页表项PTE查询到目标物理地址所处的页框号了，PTE中同样存放着1024个页框的PFN和标志位

虚拟地址与物理地址的映射

一个32位虚拟地址是如何通过分页式管理与物理内存中的一个个字节进行映射的呢？

  • 虚拟地址前十位，2 ^ 10 = 1024，决定在哪个页目录项，页目录索引
  • 虚拟地址十一到二十位，2 ^ 10 = 1024，决定在哪个页表项，此时已经拿到了对应物理地址所在的页框号PFN，页表索引
  • 虚拟地址后十二位，2 ^ 12 = 4096，页内偏移量， PFN * 4096 + 偏移量 = 具体物理地址

标志位

32位系统下，按4KB划分页框，总共有 2 ^ 20 个页框号，20 个 比特位就能表示所有的页框号，但是一个页表项用了32个比特位，剩下的比特位被用作每个页目录项的标志位，PDE和PTE同样都有标志位

  • P位/存在位：这个页框在不在物理内存里，不在就触发缺页中断
  • R/W位 / 读写位：这个页框是否可读可写
  • U/S位 / 用户/内核位：这是内核态页框还是用户态页框
  • A位 / D位：页框被访问过吗，页框是不是脏页，页面置换算法，比如clock时钟算法，全靠这两个标志位做决策

页框元数据

内存中有一个全局数组，名叫mem_map，它的成员是struct page结构体，每一个成员都是一个页框的元数据，而数组下标就是页框对应的页框号PFN，它的成员有：

  1. flags 页面状态标志
    这是一个32位/64位的无符号整数，每一位代表一个独立的状态
    • PG_locked：页框被锁住了，比如这个页框正在进行磁盘 I/O，别的进程不准动它
    • PG_dirty：这个页框是脏的，和之前标志位里的 D位有联动
    • PG_lru：这个页框当前挂在 LRU 链表上，正在参与页面置换算法的淘汰赛
    • PG_slab：这个页框被内核的 Slab 分配器剁成臊子用了
      ...
  2. _refcount 引用计数
    这个我熟，硬链接和智能指针，它记录当前有几个对象，比如进程或内核组件在使用这个页框，降为 0 就可以被操作系统回收进空闲内存池了
  3. _mapcount 映射计数
    记录有几个用户态的 PTE 指向这个页框，比如调用 fork() 之后，父子进程共享同一个页框，这个数字就会增加，实现写时拷贝 COW 就用到了这个数字
  4. lru 链表节点
    它包含两个指针 next 和 prev，用于把这个 struct page 串进各种链表里，侵入式数据结构的设计，专供页面置换算法遍历使用
  5. mapping 和 index 反向映射 RMAP / 页缓存
    记录这个页框属于哪个文件，在文件里的位置/偏移量，内存不够要从内存里踢掉这个页的时候，操作系统就能够找到所有映射了这个页的进程，改变它们的PTE，把 存在位P设为 0

所以，操作系统对物理内存的管理，就变为了对这个全局数组增删改查的过程
struct page 是给操作系统看的，PTE 和 PDE 里的标志位是给 MMU 看的，标志位是硬件层面最底层的拦截和探测标记，而页框元数据是内核管理全局内存的数据依据

内核级页目录项

操作系统开机时，操作系统本身的代码和数据会被加载进物理内存，这部分物理内存也需要用页表映射管理起来，操作系统全局存在一张内核级页表，专门用于映射 3GB ~ 4GB，0xC0000000 ~ 0xFFFFFFFF这段内核虚拟地址空间。

  1. 所有进程共享同一份内核映射，所有进程被创建时，操作系统会将全局的那份内核级页目录项，拷贝到新进程的页目录表的高地址部分，也就是说所有内核代码数据的虚拟地址都相同
  2. 内核有一块直接映射区，在这1GB内核虚拟地址中，前面的一大块，通常是896MB，这部分的映射关系，不同于用户级页表项歪七扭八的离散映射，直接映射区维持着简单的偏移量关系：内核虚拟地址 = 内核物理地址 + 0xC0000000 (3GB)，操作系统在开机初始化时，自己把这块直接映射区对应的页表项线性地填好了，这样对于内核来说，用个宏就能转换直接映射区的虚拟地址和物理地址
  3. 为什么用户态程序不能访问内核区，内核级页表的页表项中的U/S位被设置为0特权级，MMU 查表得时候发现用户态访问这个页表项直接就让 CPU 去走段错误中断门去给进程发段错误信号了

TLB

单级⻚表对连续内存要求⾼，于是引⼊了多级⻚表，但是多级⻚表也是⼀把双刃剑，在减少连续存储要求且减少存储空间的同时降低了查询效率

MMU 引入了一个叫做TLB快表的缓存结构，翻译为后备缓冲器
CPU 给 MMU 虚拟地址后，MMU 先去 TLB 那看看有没有对应的物理地址，如果有就直接拿着物理地址发到总线给内存相关硬件，不用去查页目录了
但是快表挺小的，缓存命中率难免较低，所以MMU 在页表里查到物理地址了之后会顺手将这条映射关系给 TLB，让它记录一下刷新缓存

进程切换时，因为每个进程都有独立的页表，所以进程切换通常会导致 TLB 失效，这也是多线程比多进程开销小的底层原因之一

缺页异常

设想，CPU 给 MMU 的虚拟地址，在 TLB 和⻚表都没有找到对应的物理⻚，该怎么办呢？其实这就是
缺⻚异常 Page Fault
假如⽬标内存⻚在物理内存中没有对应的物理⻚或者存在但⽆对应权限，CPU 就⽆法获取数据，这种
情况下 CPU 就会执行缺页异常的流程，缺页中断也分不同的情况，在另一篇笔记中谈到过是要让操作系统来判断缺页异常类型

  • Hard Page Fault / Major Page Fault 物理内存中没有对应的物理⻚，需要CPU打开磁盘设备读取到物理内存中，再让MMU建⽴虚拟地址和物理地址的映射
  • Soft Page Fault / Minor Page Fault 这时物理内存中是存在对应物理⻚的，只不过可能是其他进程调⼊的，发出缺⻚异常的进程不知道⽽已，此时 MMU 只需要建⽴映射即可，⽆需从磁盘读取写⼊内存，⼀般出现在多进程共享内存区域
  • Invalid Page Fault 无效缺页错误，野指针/空指针那种错误，进程访问非法内存

Note

写时拷贝也是一种缺页异常
父子进程共享物理页且都只有读权限，当其中一方尝试写入时，MMU 发现权限不足触发异常，内核介入后发现是 COW机制，于是默默分配新页框、拷贝数据并修改 PTE 权限，然后再让进程继续执行
